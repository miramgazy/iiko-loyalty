# Инструкция по развертыванию проекта на Coolify (Ubuntu 24)

Эта инструкция описывает процесс развертывания стека приложения на self-hosted платформе **Coolify** с учетом ограничений ресурсов в **512 МБ RAM** и **1 CPU**, а также проксирования через внешний порт **6054**.

---

## 1. Решенные проблемы кросс-платформенности (macOS vs Ubuntu 24)

При локальной разработке на macOS (или iOS, как упомянуто в запросе) создаются файлы и кэш, специфичные для текущей операционной системы и архитектуры (например, ARM64):
- Python виртуальное окружение в `backend/.venv` содержит скомпилированные бинарные библиотеки под Darwin.
- Папки `node_modules/` во фронтенд-приложениях содержат нативные бинарные привязки (например, для сборщика Vite/esbuild), скомпилированные под macOS.
- Локальная база данных `db.sqlite3` и временные файлы.

Если при сборке Docker-образа в Coolify на Ubuntu 24 скопировать эти папки внутрь контейнера, сборка либо завершится ошибкой, либо контейнер будет падать при запуске из-за несовместимости бинарных форматов (`Exec format error`).

**Как это решено:**
В корень проекта добавлены файлы:
1. `backend/.dockerignore` — исключает копирование `.venv`, `db.sqlite3`, `staticfiles`, `media` и `__pycache__`.
2. `frontend/.dockerignore` — исключает копирование папок `node_modules` и `dist` для панели администратора (`admin/`) и Telegram Mini App (`tma/`).

Это гарантирует, что Docker на Ubuntu 24 скачает чистые зависимости и скомпилирует их непосредственно под архитектуру целевого сервера.

## 2. Распределение лимитов ресурсов (512 MB RAM / 1 CPU)

Поскольку весь стек состоит из 6 контейнеров, запуск их без ограничений на сервере с 512 МБ памяти гарантированно приведет к падению системы по OOM (Out Of Memory). Для обеспечения стабильности мы применили **Strict Service-Level Limits** в файле `docker-compose.coolify.yml`.

Лимиты распределены следующим образом:

| Сервис | Описание | Ограничение памяти | Ограничение CPU | Оптимизации |
| :--- | :--- | :--- | :--- | :--- |
| **db** | PostgreSQL 15 | **128 MB** | **0.2 CPU** | Оптимально для небольших баз данных. |
| **redis** | Redis 7 Cache/Broker | **64 MB** | **0.1 CPU** | Легковесный in-memory кэш. |
| **backend** | Django Daphne (ASGI) | **128 MB** | **0.3 CPU** | Достаточно для обработки ASGI/WebSocket запросов. |
| **celery-worker** | Celery фоновые задачи | **96 MB** | **0.2 CPU** | Запуск с параметром `--concurrency=1` для снижения footprint Django. |
| **celery-beat** | Celery планировщик | **48 MB** | **0.1 CPU** | Минимальный планировщик. |
| **nginx** | Nginx Proxy / Frontend | **48 MB** | **0.1 CPU** | Раздача статики и проксирование API. |
| **Итого** | **Весь стек** | **512 MB** | **1.0 CPU** | **Полное соответствие системному лимиту.** |

> [!TIP]
> Если вы решите развернуть стек на более мощном сервере, вы можете пропорционально увеличить лимиты в секциях `deploy.resources.limits` внутри файла [docker-compose.coolify.yml](file:///Users/miramgazy/Projects/iiko-loyal/docker-compose.coolify.yml).

## 3. Настройка внешнего порта 6054 и проксирования

В сервисе `nginx` настроен проброс портов:
```yaml
    ports:
      - "6054:80"
```
Это означает, что Nginx будет слушать порт `6054` на хост-машине VPS под управлением Ubuntu 24.

Вы можете настроить проксирование двумя способами:

### Способ А: Внешний Nginx/Caddy/Cloudflare Tunnel на хост-машине
Если у вас уже установлен Nginx или Caddy на самой системе Ubuntu 24, вы можете настроить проксирование с вашего домена на локальный порт `6054`:
*Пример для Nginx на хосте:*
```nginx
server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://127.0.0.1:6054;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### Способ Б: Встроенный обратный прокси Coolify (Traefik / Caddy)
Если вы хотите использовать автоматический SSL от Coolify:
1. В настройках сервиса `nginx` в Coolify укажите ваш домен (например, `https://your-domain.com`).
2. Coolify автоматически направит трафик на внутренний порт `80` контейнера `nginx`. При этом порт `6054` всё равно будет открыт на хосте для резервного/прямого доступа.

## 4. Пошаговый процесс деплоя в Coolify

1. **Запушьте изменения в Git**:
   Убедитесь, что файлы `.dockerignore` и `docker-compose.coolify.yml` находятся в вашей основной ветке.

2. **Создайте проект в Coolify**:
   - Перейдите в панель Coolify.
   - Нажмите **Projects** -> **Create New Project**.
   - Выберите созданный проект и нажмите **+ New Resource**.

3. **Выберите источник Docker Compose**:
   - Выберите **Docker Compose**.
   - Укажите тип источника: **Git Repository**.
   - Выберите ваш Git-интегратор (GitHub/GitLab) или укажите ссылку на публичный/приватный репозиторий.
   - Выберите ветку для деплоя.

4. **Укажите путь к конфигурационному файлу**:
   - По умолчанию Coolify ищет `docker-compose.yml`.
   - В поле **Compose File Path** измените путь на:
     `docker-compose.coolify.yml`

5. **Настройте переменные окружения (`Environment Variables`)**:
   - Перейдите во вкладку **Environment Variables** в настройках ресурса Coolify.
   - Скопируйте переменные из вашего `.env` файла (или используйте `.env.example` как референс).
   - Обязательно укажите:
     - `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` — для базы данных.
     - `DJANGO_SECRET_KEY` — случайная строка.
     - `DJANGO_DEBUG=0` — отключение дебага на проде.
     - `DJANGO_ALLOWED_HOSTS` — домен вашего приложения и имя хоста `backend`.
     - `DJANGO_CSRF_TRUSTED_ORIGINS` — URL с `https://` для вашего домена (чтобы избежать CSRF-ошибок при работе через прокси).
     - `REDIS_URL=redis://redis:6379/0`
     - `FIELD_ENCRYPTION_KEY` — 32-байтовый Fernet ключ.
     - `WEBHOOK_DOMAIN` — домен для вебхуков Telegram.
     - `VITE_WEBHOOK_DOMAIN` — домен для сборки фронтенда (передается в качестве build-arg в nginx).
     - `VITE_API_BASE_URL` — ссылка на API бэкенда (например, `https://your-domain.com/api`).

6. **Запустите деплой**:
   - Нажмите кнопку **Deploy**.
   - Coolify автоматически клонирует репозиторий, выполнит сборку образов согласно `.dockerignore` и запустит контейнеры на Ubuntu 24.
   - Вы можете следить за логами сборки прямо в интерфейсе Coolify.
