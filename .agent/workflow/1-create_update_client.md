### Контекст проекта
Мы реализуем интеграционный слой с API автоматизации ресторанов iiko (iiko Cloud API / iiko_transport и legacy iikoCard/iiko.biz) для мультитенантной SaaS-платформы лояльности. 
Стек: Python 3.11+, Django 4.2+, Django REST Framework, PostgreSQL, Redis (для кэширования токенов).

### Архитектурная задача
Необходимо разработать отказоустойчивый сервисный слой на бэкенде, который изолированно для каждого тенанта (`Organization`) умеет работать с системой лояльности iiko. 
Так как типы интеграций у ресторанов могут отличаться (iiko Cloud API или старый iikoCard API), архитектура должна использовать паттерн «Стратегия» (Strategy) или адаптер, инкапсулирующий вызовы к внешним серверам.

### ТРЕБОВАНИЯ К РЕАЛИЗАЦИИ

#### 1. Логика кэширования и переиспользования токенов (Шаг 1)
У каждой организации в БД есть свой `iiko_api_login`. Для выполнения запросов нужно получать временный `token`.
- **Эндпоинт iiko Cloud:** `https://api-ru.iiko.services/api/1/loyalty/iiko/customer/access_token`
- **Логика хранения:** Токен выдается на 30 минут. Бэкенд должен кэшировать его в Redis или в локальной таблице `IikoTokens` (привязанной к `Organization`). 
- Перед каждым запросом к iiko сервис обязан проверить: если текущий токен существует и с момента его получения прошло МЕНЕЕ 25 минут (берем с запасом), использовать его. Если токен просрочен или отсутствует — отправить POST-запрос с `apiLogin`, сохранить новый токен и метку времени, а затем продолжить флоу.

#### 2. Расширение моделей данных (PostgreSQL)
Необходимо предусмотреть поля в моделях для сохранения критически важных данных, возвращаемых iiko.

**Модель `Organization`:**
- `iiko_integration_type`: Выбор (iiko_transport, iiko_card). От этого зависит базовый URL и формат эндпоинтов.

**Модель `Customer`:**
- `iiko_customer_id` (UUIDField, индексируемый) — сохраняется из ответа `id` на Шаге 2 или Шаге 4.
- `iiko_card_number` (CharField, max_length=50) — номер привязанной карты (Шаг 3).
- `iiko_card_id` (UUIDField, null=True) — ID карты из массива `cards` внутри iiko.
- `iiko_categories` (JSONField) — массив категорий пользователя (например, `[{"id": "...", "name": "new_user"}]`).
- `loyalty_balance` (DecimalField, max_digits=10, decimal_places=2, default=0.00) — текущий кэш баланса из кошелька лояльности iiko.

#### 3. Сервисный слой интеграции (Интерфейс iiko API)
Реализовать класс `IikoIntegrationService`, который принимает объект `organization` и предоставляет следующие асинхронные/синхронные методы (используя `httpx`):

##### А. Создание / Обновление пользователя (Шаг 2)
- **Эндпоинт:** `POST /api/1/loyalty/iiko/customer/create_or_update`
- **Параметры:** `organizationId`, `phone`, `name`, `surname`, `email`, `birthday` (в формате ISO 8601: `"YYYY-MM-DDTHH:MM:SSZ"`), `sex`, `consentStatus: 1`.
- **Поведение:** Отправляет данные гостя. Полученный из ответа UUID (`id`) записывает в поле `customer.iiko_customer_id` соответствующего клиента в нашей БД.

##### Б. Создание/Привязка виртуальной карты (Шаг 3)
- **Эндпоинт:** `POST /api/1/loyalty/iiko/customer/card/add`
- **Параметры:** `organizationId`, `customerId` (наш `iiko_customer_id`), `cardNumber` и `cardTrack`.
- **Поведение:** В качестве номера карты и трека использовать нормализованный номер телефона клиента БЕЗ знака плюс (или генерировать уникальный пул номеров, согласованный в настройках тенанта). После успешного ответа `200 OK` (возвращает `{}`), зафиксировать номер карты в БД.

##### В. Получение полной информации о клиенте (Шаг 4)
- **Эндпоинт:** `POST /api/1/loyalty/iiko/customer/info`
- **Параметры:** `organizationId`, `type: "phone"`, `phone`.
- **Поведение:** Самый часто вызываемый метод (pull-запрос баланса). Из ответа бэкенд обязан:
  1. Вытащить баланс из массива `walletBalances` (найти кошелек, обычно первый или с типом 1) и обновить поле `customer.loyalty_balance`.
  2. Обновить массив `customer.iiko_categories` на основе актуальных данных из iiko.
  3. Проверить массив `cards`. Если карта в iiko есть, а у нас в БД нет — синхронизировать (записать номер и `id` карты).
