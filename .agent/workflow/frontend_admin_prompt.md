# Техническое задание: Разработка Frontend Панели Управления (Vue 3 + Vite + Tailwind CSS)

## 1. Контекст и архитектура
- **Назначение**: Панель управления (Admin Panel) мультитенантной SaaS-платформы лояльности для ресторанного бизнеса.
- **Стек**: Vue 3 (Composition API), Vite, Tailwind CSS, Pinia, Axios, Vue Router.
- **Целевая аудитория**:
  - `SuperAdmin` / `SuperUser` — глобальные администраторы SaaS-платформы.
  - `OrgManager` — владельцы/управляющие конкретных ресторанных сетей (тенантов).
  - `OrgAdmin` — администраторы на местах (в ресторанах).
- **Авторизация**: Классический email/password + JWT. Токены (Access и Refresh) сохраняются в `localStorage` или в защищенных cookies. Axios-интерцептор автоматически подставляет заголовок `Authorization: Bearer <token>` и выполняет обновление токена (Refresh Flow) при ошибке 401.

---

## 2. Ролевое разграничение (RBAC) на клиенте
Реализуй маршрутизатор `Vue Router` с защищенными роутами (`Navigation Guards`).
- В Pinia store (`authStore`) хранится текущий пользователь, список его членств в организациях (`UserOrganization` с ролями) и выбранная в данный момент активная организация (`current_organization_id`).
- При переходе на роуты с префиксом `/superadmin/*` guard должен проверять, является ли роль пользователя `SuperUser` или `SuperAdmin`.
- При переходе на роуты с префиксом `/admin/*` guard должен проверять наличие роли `OrgManager` или `OrgAdmin` для выбранной в данный момент организации.

## 3. Основные экраны и функционал

### 3.1. Страница авторизации (`/login`)
Страница входа должна быть выполнена в виде двух вкладок (Tabs):
1. **Вкладка «Имя пользователя или Email»**:
   - Поля ввода: `username_or_email` (Имя пользователя или Email) и `password` (Пароль).
   - При отправке шлет POST-запрос на бэкенд с телом `{ "username": username_or_email, "password": password }`.
2. **Вкладка «Номер телефона»**:
   - Поля ввода: `phone` (поле с маской ввода номера телефона, например `+7 (7XX) XXX-XX-XX`) и `password` (Пароль).
   - При отправке автоматически удаляет форматирование телефона до формата `+7XXXXXXXXXX` и шлет POST-запрос с телом `{ "username": clean_phone, "password": password }`.

После успешного входа в `authStore` записываются JWT-токены, роль пользователя и список доступных ему организаций. Если у пользователя есть доступ к нескольким организациям, перед перенаправлением на основной дашборд отображается промежуточный экран выбора тенанта (организации).

### 3.2. Глобальный кабинет SuperAdmin (`/superadmin`)
Доступен только глобальным администраторам системы.
- **Список организаций (тенантов)**:
  - Таблица со списком всех ресторанных сетей в системе (название, slug, статус `is_active`, дата создания).
  - Возможность создать новую организацию (форма: Название, Slug, email владельца).
  - Кнопки приостановки/активации организации (`is_active = true/false`).
- **Управление глобальными тарифами/настройками**: (заглушка).

### 3.3. Настройки Организации (`/admin/settings`)
Доступно роли `OrgManager`. Форма разделена на три вкладки:

1. **Основные и Telegram-настройки**:
   - Поля ввода: Название ресторана, Slug (read-only), Адрес.
   - Настройки бота: `tg_bot_token` (скрытый по умолчанию с кнопкой «показать»), `tg_bot_username` (без символа `@`), `tma_name`.
   - Кнопка «Сохранить».
2. **Интеграция с iiko Cloud**:
   - Поля ввода: `iiko_api_base_url` (с предзаполненным дефолтом `https://api-ru.iiko.services/api/1`), `iiko_api_login` (поле пароля/токена), `iiko_organization_id` (UUID), `iiko_loyalty_program_id` (UUID).
3. **Брендинг Telegram Mini App (TMA Customization)**:
   - Цветовая тема: Удобный Color Picker (выбор HEX-кода основного брендового цвета организации, записывается в поле `design_color` в JSON-объект `branding`).
   - Приветственный текст: Поле ввода `greeting_text` (по умолчанию "Добро пожаловать!").
   - Логотип: Компонент для загрузки изображения (`logo`), с предпросмотром.

### 3.4. Управление персоналом организации (`/admin/employees`)
Доступно для `OrgManager`.
- Список сотрудников, привязанных к текущей организации (ФИО, Email, роль: `OrgManager` или `OrgAdmin`).
- Форма приглашения/создания нового сотрудника (Email, ФИО, выбор роли).
- Возможность отозвать доступ (удалить связь `UserOrganization`).

### 3.5. База клиентов лояльности (`/admin/customers`)
Доступно для `OrgManager` и `OrgAdmin`.
- Таблица клиентов, зарегистрированных через Mini App в данной организации:
  - Имя, Фамилия, Номер телефона (с форматированием), Telegram ID, Дата регистрации, Статус интеграции iiko (`iiko_customer_id` или индикатор успешной синхронизации).
- Фильтрация: поиск по телефону, имени и фамилии.
- Экспорт списка клиентов в CSV/Excel (заглушка).

## 4. UI/UX Требования
- Использовать современный адаптивный дизайн на Tailwind CSS.
- Компоненты форм должны содержать четкую валидацию (например, маска ввода для телефонов, UUID-валидатор для полей iiko, HEX-валидатор для цветов).
- Обработка состояний загрузки (Loading Spinners) на всех кнопках сохранения и таблицах.
- Всплывающие уведомления (Toast Notifications) при успешных операциях и ошибках API.
