Промт для ИИ-разработчика: Реализация Webhook-обработчика iiko Cloud и Push-уведомлений в Telegram
Markdown
### Контекст проекта
В нашей мультитенантной SaaS-платформе лояльности настроена связь с iiko Cloud API: мы умеем создавать пользователей и запрашивать баланс. 
Текущая задача: Реализовать обработку входящих вебхуков от iiko Cloud об изменении баланса клиента и настроить автоматическую отправку push-сообщений в Telegram через бота соответствующего ресторана.

### Архитектурный флоу
iiko Cloud Event ──> Django Webhook View ──> Нахождение Tenant/Customer ──> Celery Task ──> Telegram Bot API ──> Гость

---

### ТРЕБОВАНИЯ К РЕАЛИЗАЦИИ

#### 1. Эндпоинт для приема вебхуков от iiko (Django API)
Создать публичный URL `/api/v1/integrations/iiko/webhook/` (без `permission_classes`, так как запросы идут от серверов iiko).
- **Безопасность (Опционально, но рекомендуется):** Предусмотреть валидацию IP-адресов iiko или проверку секретного токена в заголовках, если это настроено в личном кабинете iiko.
- **Поведение:** Эндпоинт должен максимально быстро ответить `200 OK` серверу iiko, чтобы избежать таймаутов, а саму обработку тела запроса передать асинхронно в **Celery-задачу**.

#### 2. Celery-задача обработки события лояльности
Реализовать задачу `process_iiko_webhook(payload)`. Согласно документации iiko Cloud, при изменении баланса приходит событие типа `LoyaltyCustomerWalletBalanceChanged` или структура уведомления лояльности.

**Логика парсинга payload:**
1. Извлечь `organizationId` и `customerId` (или номер телефона/карты) из тела вебхука.
2. Извлечь параметры изменения: `walletName`, `balance` (новый баланс), `changeAmount` (сумма изменения: начисление или списание).
3. По базе данных PostgreSQL найти клиента:
   
```python
   customer = Customer.objects.select_related('organization').get(
       organization__iiko_organization_id=organization_id,
       iiko_customer_id=customer_id
   )
Обновить текущий кэш баланса в нашей базе данных: customer.loyalty_balance = balance.
Отправить событие обновления в WebSocket-канал пользователя (Django Channels), чтобы на фронтенде в Vue Mini App баланс обновился в реальном времени без перезагрузки.
3. Сервис отправки Push-сообщений через Telegram Bot API
После обновления БД, внутри Celery-задачи, инициировать отправку сообщения.
Бэкенд должен использовать tg_bot_token именно той Organization, к которой привязан клиент. Использование глобального или захардкоженного токена запрещено!
Сформировать красивое, клиентоориентированное сообщение.
Пример логики формирования текста:
Если changeAmount > 0:
«🔥 Вам начислено {changeAmount} бонусов в заведении {organization.name}! Ваш текущий баланс: {balance} бонусов. Ждем вас снова! 🥰»
Если changeAmount < 0:
«💳 Списание бонусов: {abs(changeAmount)} б. в {organization.name}. Остаток на карте: {balance} бонусов. Спасибо за визит! ✨»
Внизу сообщения обязательно прикрепить Inline-кнопку (InlineKeyboardMarkup) со ссылкой на Telegram Mini App этого заведения (WebAppInfo), чтобы пользователь мог в один клик открыть свою карту и увидеть обновленный QR-код.
4. Обработка ошибок и логирование
Реализовать модель IikoWebhookLog для логирования всех входящих запросов от iiko (дата, тип события, raw JSON payload, статус обработки SUCCESS/FAILED). Это критически важно для разбора инцидентов, если у клиента "не дошли бонусы".
Обернуть поиск пользователей в блоки try/except Customer.DoesNotExist, чтобы битые или тестовые вебхуки от iiko не ломали очередь Celery.

Модель IikoWebhookLog для истории логов.
