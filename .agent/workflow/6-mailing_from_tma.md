# Системный промт: Реализация Telegram-рассылок и сбора согласий пользователей (TMA + Django + Celery + Vue)

Вы — опытный full-stack разработчик. Ваша задача — перенести и реализовать функционал **получения разрешений на рассылку** и **проведения целевых рассылок** в Telegram-боте из референсного проекта.

Функционал включает в себя три основные части:
1. **Django Backend**: модели данных, API-эндпоинты для вебхуков Telegram, ручная настройка вебхуков, API управления рассылками.
2. **Celery Tasks**: периодическая проверка рассылок и асинхронная отправка сообщений батчами с соблюдением лимитов Telegram API (30 сообщений/сек) и автоматической отпиской заблокировавших бота пользователей.
3. **Frontend (Vite/Vue.js)**:
   - **TMA (Telegram Mini App)**: Обязательный шаг онбординга (запрос согласия на рассылку при первом входе, если выбор `is_bot_subscribed` еще не сделан — равен `null`).
   - **Панель Администратора**: Интерфейс создания рассылок (название, сегментация аудитории, выбор времени отправки, шаблоны на двух языках RU/KZ с поддержкой плейсхолдеров, прекалькуляция охвата, тестовая отправка на Telegram ID администратора).

## 1. Спецификация Backend (Django REST Framework)

### 1.1. База Данных и Модели

#### Обновление модели Пользователя (`User` / `accounts`)
Добавьте поля в модель пользователя:
- `telegram_id` (BigIntegerField, null=True, unique для организации) — ID пользователя в Telegram.
- `is_bot_subscribed` (BooleanField, null=True, blank=True) — согласие на получение рассылок. 
  - *Важно*: значение по умолчанию должно быть `None` (`null=True`), что означает "выбор еще не сделан" и триггерит окно онбординга. `True` — согласился, `False` — отказался / заблокировал бота.
- `language` (CharField, default='ru') — язык пользователя ('ru' или 'kz').

#### Создание модели Рассылки (`MailingTask` / `mailing`)
Создайте модель `MailingTask` со следующей структурой:
- `organization` (ForeignKey к `Organization`, on_delete=CASCADE)
- `title` (CharField, max_length=255) — название кампании.
- `message_ru` (TextField) — текст сообщения на русском. Поддерживает плейсхолдер `{{user_name}}`.
- `message_kz` (TextField) — текст сообщения на казахском. Поддерживает плейсхолдер `{{user_name}}`.
- `scheduled_at` (DateTimeField) — запланированное время старта.
- `audience_type` (CharField, choices: `all` — все, `active` — активные за 60 дней, `inactive` — неактивные более 60 дней)
- `status` (CharField, default=`draft`, choices: `draft`, `scheduled`, `in_progress`, `done`, `error`)
- **Поля аналитики**:
  - `total_recipients` (PositiveIntegerField, default=0) — размер целевой аудитории при старте.
  - `sent_success` (PositiveIntegerField, default=0) — успешно отправлено.
  - `failed_count` (PositiveIntegerField, default=0) — ошибок отправки (сетевые проблемы и др.).
  - `unsubscribed_count` (PositiveIntegerField, default=0) — количество пользователей, заблокировавших бота (ошибка 403).
- **Курсор пагинации**:
  - `last_processed_user_id` (IntegerField, default=0) — ID последнего успешно обработанного пользователя для безопасного продолжения рассылки батчами.

### 1.2. API-Эндпоинты

#### 1. TMA Webhook (`TmaWebhookView`)
Эндпоинт: `/api/accounts/tma/webhook/<slug:org_slug>/<str:token>/` (Доступ: `AllowAny`)
- Слушает обновления от Telegram Webhook (`message`, `callback_query`).
- Должен уметь обрабатывать команду `/getmyid` в личных сообщениях (возвращает Telegram ID пользователя текстом `Ваш Telegram ID: <code>ID</code>`).
- **Сбор согласий через Callback Queries**:
  - Если приходит `callback_query` с `data`, начинающимся с `consent_`:
    - Шаблон `data`: `consent_yes_<user_id>` или `consent_no_<user_id>`.
    - Обновляет у пользователя с `id=user_id` поле `is_bot_subscribed` на `True` или `False`.
    - Отправляет ответное сообщение через Telegram API: «Благодарим за доверие!» или «Хорошо, мы не будем беспокоить вас рассылками.» с инлайн-кнопкой «Вернуться в приложение» (ссылка на TMA).
    - Вызывает метод `answerCallbackQuery` для скрытия лоадера кнопки в интерфейсе Telegram.

#### 2. Настройка Вебхука (`SetWebhookView`)
Эндпоинт: `/api/organization/set-webhook/` (Доступ: `IsAuthenticated`, только владелец/администратор)
- Отправляет запрос в Telegram API `setWebhook?url=<webhook_url>`, где `webhook_url` формируется динамически: `{WEBHOOK_DOMAIN}/api/accounts/tma/webhook/{org_slug}/{bot_token}/`.

#### 3. Набор эндпоинтов рассылки (`MailingTaskViewSet`)
Эндпоинт: `/api/mailings/` (Доступ: `IsAuthenticated`, фильтрация по `request.user.organization`)
- Стандартный CRUD для рассылок (при создании автоматически выставляется организация текущего пользователя и статус `scheduled`).
- **Кастомный экшен `count_recipients` (GET)**:
  - Возвращает число `{ count: N }` получателей для выбранного `audience_type`.
  - Правила фильтрации получателей:
    - Базовая выборка: Пользователи организации, у которых `is_bot_subscribed=True` и `telegram_id` заполнен.
    - `all`: без дополнительных условий.
    - `active`: пользователи, у которых есть визиты (записи) в статусе "Подтвержден/Завершен" за последние 60 дней.
    - `inactive`: пользователи, у которых нет ни одного визита за последние 60 дней.
- **Кастомный экшен `send_test` (POST)**:
  - Принимает `{ telegram_id: "..." }`.
  - Отправляет тестовое сообщение на этот ID с припиской `[ТЕСТОВОЕ СООБЩЕНИЕ]`. Шаблон сообщения выбирается по языку текущего авторизованного администратора (`request.user.language`), плейсхолдер `{{user_name}}` заменяется на имя администратора.

### 1.3. Интеграция с Telegram API (Utility функции)
Реализуйте функции отправки сообщений с логированием ошибок:
- `send_telegram_message(bot_token, chat_id, text, reply_markup=None)`: отправка сообщений по HTTP POST на `https://api.telegram.org/bot{token}/sendMessage` с `parse_mode='HTML'`.
- Возвращайте словарь с результатом `{ success: bool, status_code: int, response: dict }` для точной обработки ошибок на уровне Celery-воркера.

## 2. Асинхронные Задачи Celery

Для стабильной работы рассылок без блокировки основного потока API и обхода лимитов Telegram API (не более 30 сообщений в секунду суммарно от одного бота) используйте следующую архитектуру Celery:

### 2.1. Периодический Диспетчер (`check_scheduled_mailings`)
- Запускается раз в 10 минут (через Celery Beat / Cron).
- Находит все `MailingTask` в статусе `scheduled` или `in_progress`, у которых запланированное время отправки меньше или равно текущему времени (`scheduled_at <= now()`).
- Для каждой найденной рассылки запускает воркер: `process_mailing_batch.delay(task_id)`.

### 2.2. Потоковый Воркер Батчей (`process_mailing_batch`)
- Должен работать в транзакции базы данных с блокировкой строки рассылки (`select_for_update()`), чтобы избежать параллельного запуска одного и того же батча.
- **Инициализация**: При первом запуске (статус `scheduled`) вычисляет общее число получателей (`total_recipients`), сохраняет его и переводит статус в `in_progress`.
- **Выборка батча**: Выбирает из отфильтрованного QuerySet получателей очередную порцию — **строго до 30 пользователей** — по условию `id > task.last_processed_user_id` с сортировкой по `id` (курсорная пагинация).
- **Цикл отправки**:
  - Для каждого пользователя из батча:
    - Выбирает шаблон текста на основе `user.language` (KZ/RU).
    - Заменяет `{{user_name}}` на `user.first_name` или значение по умолчанию (например, "Клиент").
    - Вызывает `send_telegram_message`.
    - **Обработка ответа**:
      - Если HTTP статус `403 (Forbidden)` (пользователь заблокировал бота) -> меняет у пользователя `is_bot_subscribed` на `False` и увеличивает `unsubscribed_count` рассылки на 1.
      - Если отправка успешна -> увеличивает `sent_success` на 1.
      - В противном случае -> увеличивает `failed_count` на 1.
      - Обновляет `last_processed_user_id` до текущего `user.id`.
- **Завершение или перезапуск**:
  - Сохраняет обновленные счетчики и курсор в БД.
  - Если длина текущего батча была равна 30 -> в БД могут быть еще получатели. Планирует повторный асинхронный запуск этой же задачи с задержкой в 1 секунду: `process_mailing_batch.apply_async(args=[task_id], countdown=1)`. Задержка в 1 секунду гарантирует соблюдение лимитов Telegram API.
  - Если длина батча меньше 30 -> рассылка завершена. Статус переводится в `done`.

## 3. Фронтенд Часть (Vue.js 3 / Pinia)

### 3.1. Telegram Mini App (TMA Client)

#### Интеграция Onboarding Guard в `App.vue` / Router
- В Pinia-сторе авторизации (`auth.js`) определите геттер:
  `needsConsent: (state) => state.user?.is_bot_subscribed == null`
- При инициализации приложения после авторизации пользователя через `initData`:
  - Проверьте, если `auth.needsConsent` истинно -> перенаправьте пользователя на страницу согласия: `/onboarding/consent`.
  - Дополнительно защитите пути роутера с помощью `beforeEach` гварда.

#### Экран запроса согласия (`BotConsentView.vue`)
Красивый, современный экран онбординга:
- Заголовок, описание преимуществ рассылки (акции, скидки, напоминания о записи).
- Две кнопки взаимодействия:
  - **Разрешить** (`handleAllow`): Отправляет PATCH-запрос на обновление профиля пользователя `/api/accounts/tma/me/` с `{ is_bot_subscribed: true }`. После успешного сохранения перенаправляет на главный экран.
  - **Отклонить** (`handleDecline`): Отправляет PATCH-запрос с `{ is_bot_subscribed: false }`. Перенаправляет на главный экран.
- Во время сохранения кнопки должны переходить в состояние `loading`/`disabled`.

### 3.2. Панель Администратора (`MailingsView.vue`)

Разработайте полноценный интерфейс управления рассылками.

#### 1. Список рассылок (Таблица)
- Колонки: Название рассылки, Целевой сегмент, Дата/время отправки, Статус (Черновик, Запланирована, В процессе, Завершена, Ошибка), Прогресс (процент отправки, количество успешных, ошибок, отписок в реальном времени).
- Кнопки действий:
  - Тестовая отправка (доступна для всех статусов).
  - Удаление (доступно только для `scheduled` или `draft`).

#### 2. Модальное окно создания рассылки
- **Название**: Текстовый инпут.
- **Сегмент аудитории**: Выпадающий список (`all`, `active`, `inactive`). 
  - При изменении сегмента отправляется запрос на `/api/mailings/count_recipients/?audience_type=...` и отображается плашка: «Целевая аудитория: N человек».
- **Время отправки**: DateTime picker (по умолчанию устанавливайте текущее время + 10 минут).
- **Шаблоны сообщений**:
  - Поля ввода текста отдельно для Русского и Казахского языков.
  - Удобная кнопка-утилита «Вставить {{user_name}}», которая добавляет плейсхолдер имени в текущую позицию курсора активного текстового поля.
- Валидация полей перед отправкой.

#### 3. Модальное окно тестовой отправки
- Спрашивает у администратора его `telegram_id` (при наличии в профиле подставляет автоматически).
- Содержит подсказку: «Узнать свой ID можно, написав /getmyid боту».
- По кнопке «Отправить тест» отправляет POST-запрос на `/api/mailings/{id}/send_test/` и выводит уведомление об успехе.

## Требования к качеству кода
1. **Атомарность транзакций**: Все операции с батчем рассылок в Celery должны выполняться в контексте `with transaction.atomic()` с явной блокировкой строки рассылки `select_for_update()`.
2. **Безопасность курсора**: Не используйте `OFFSET` для пагинации рассылок. Используйте только курсор `id__gt=last_processed_user_id` для предотвращения дублирования отправки и пропуска адресатов при изменении базы данных в процессе рассылки.
3. **Throttling (Ограничение скорости)**: Строго соблюдайте задержку `countdown=1` между батчами по 30 пользователей для исключения бана бота со стороны Telegram.
4. **Локализация и UX**: Адаптируйте интерфейс TMA под цветовую схему Telegram (`window.Telegram.WebApp.themeParams`).
5. **Безопасное форматирование**: Использовать `parse_mode='HTML'`. Тексты сообщений рассылки должны проходить базовое экранирование HTML тегов, если они содержат пользовательский ввод (но разрешать теги форматирования `<b>`, `<i>`, `<code>`, внесенные администратором).
